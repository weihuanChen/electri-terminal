# Internationalization Governance Overview

## 1. Purpose

This document defines the global plan for multilingual localization.

The goal is not to translate the website mechanically. The goal is to build a controlled multilingual publishing system where high-value pages are localized well, lower-risk content can be produced efficiently, and no unfinished language URLs are exposed to Google Search Console.

The core priority is:

1. Search integrity
2. Conversion quality
3. Localization quality
4. Cost and production speed

If these priorities conflict, search integrity wins. A language can stay unpublished longer, but dirty URLs must not enter sitemap, hreflang, navigation, or GSC submission.

---

## 2. Non-Negotiable Rules

### 2.1 No dirty links

No language release is allowed if it creates dirty links.

Dirty links include:

- URLs that return 404/500 but are referenced by sitemap, hreflang, navigation, structured data, or internal links.
- URLs that redirect but are still submitted as canonical sitemap URLs.
- URLs that are marked `noindex` but appear in sitemap.
- hreflang targets that are not published.
- hreflang clusters that are not reciprocal.
- localized URLs that display untranslated English body content as the primary page content.
- localized article internal links pointing to missing localized article targets.
- canonical URLs pointing to another language version when the localized page is meant to be indexable.

The detailed gate is defined in `docs/i18n/99-gsc-link-integrity-gate.md`.

### 2.2 Published means index-ready

`published` must mean the page is safe for search engines.

Preview, draft, machine-generated, and under-review content must not be included in:

- sitemap
- hreflang alternates
- language switcher targets
- public navigation
- automated GSC submission

### 2.3 No automatic English fallback on localized URLs

If `/ru/...` does not have a valid Russian version, it must not silently render the English page as indexable content.

Allowed behavior:

- return 404 for public users;
- return a preview-only page with `noindex` for authenticated admins;
- link to the English original only as an intentional user action, not as the localized canonical page.

### 2.4 Content ID is identity; slug is presentation

Localization must be based on stable entity IDs, not only slugs.

Products, categories, product families, articles, and static pages need a stable source identity. Slugs can change or be localized later, but the translation relationship must remain attached to the source entity.

## 3. Recommended Architecture Direction

### 3.1 Routing

Recommended route model:

- English default language keeps current root URLs, such as `/products/uk-2-5-terminal-block`.
- Other languages use a locale prefix, such as `/ru/products/uk-2-5-terminal-block`.

This avoids a full English URL migration while giving every non-English language a stable URL namespace.

Future alternative:

- all languages use prefixes, such as `/en/...` and `/ru/...`.

This is cleaner long term but requires larger redirect and migration work. It should not be the first implementation path unless there is a strong SEO migration reason.

### 3.2 Framework direction

The project is a Next.js App Router project. The stable implementation direction is:

- introduce a locale segment, conceptually `app/[locale]/...`;
- centralize locale definitions and route generation;
- generate metadata, canonical, sitemap, hreflang, breadcrumbs, JSON-LD, navigation, and internal links from one resolver layer.

`next-intl` is the preferred community option to evaluate first because it supports App Router routing configuration, locale prefixes, localized pathnames, and centralized locale handling.

### 3.3 URL resolver as a core service

All public URL generation must go through one resolver.

Required resolver outputs:

- canonical URL
- localized URL
- x-default URL
- hreflang cluster
- sitemap eligibility
- language switcher targets
- internal link target
- redirect target for legacy slugs

No component should manually compose localized public URLs after P1.

## 4. Language Release Model

### 4.1 Language-level states

Each language should have a global state:

- `draft`: internal planning only.
- `prelaunch`: content can be previewed and audited, but not indexed.
- `published`: language can appear in sitemap, hreflang, language switcher, and public navigation.
- `paused`: language remains in the database but is temporarily removed from search exposure.

Only `published` languages can be submitted to GSC.

### 4.2 Page-level localization states

Each localized page should have its own state:

- `missing`: no localized version exists.
- `draft`: translation started but incomplete.
- `machine_ready`: generated by automation and structurally valid.
- `review_required`: waiting for human review.
- `approved`: content approved but not yet exposed to search.
- `published`: index-ready.
- `stale`: source English content changed after translation.

Only `published` page translations can appear in sitemap and hreflang.

### 4.3 Stale handling

When the English source changes, affected translations must be marked `stale` unless the changed fields are explicitly declared non-localizable.

Stale pages should not automatically be removed from search if the old translation is still accurate enough, but they must appear in the admin queue and be blocked from new release certification until reviewed.

## 5. Page Classes

### 5.1 L1 - Core Static Pages

Examples:

- homepage
- contact
- manufacturing
- selection-guide
- resources
- quality/certification pages
- future high-value pages such as solutions

Characteristics:

- low quantity
- high SEO and trust value
- high localization requirement
- human review required

Release rule:

- A language cannot be published unless all required L1 pages are published for that language.

### 5.2 L2 - Catalog and Conversion Pages

Examples:

- product pages
- category pages
- product family pages
- product listing pages

Characteristics:

- high quantity
- conversion-critical
- structured data heavy
- terminology-sensitive
- suitable for automation with strict validation

Release rule:

- A language cannot be published unless required L2 catalog coverage passes the configured threshold.
- Any page exposed through navigation, listing, breadcrumb, related products, or structured data must have a valid localized target.

Ongoing rule after language launch:

- New products, categories, or families added after a language is already published do not automatically become published in that language.
- A new English product can be published while its French/Russian/German translations are still missing, but the product must be excluded from those localized URL graphs.
- Until the localized product translation is `published`, it must not appear in localized sitemap, hreflang, language switchers, product listing pages, family pages, category pages, related-product modules, search results, breadcrumbs, or JSON-LD.
- This is an entity-level exposure gate, not a reason to unpublish the whole language.

### 5.3 L3 - Articles, Applications, and Guides

Examples:

- blog articles
- guides
- application pages
- FAQs

Characteristics:

- medium quantity
- lower direct conversion weight
- long-form content
- acceptable to publish partially by language
- internal links are the main risk

Release rule:

- Missing localized articles are acceptable.
- Published localized articles must pass internal link validation.
- Internal links may only point to published same-language targets or approved same-language neutral hub pages.
- Cross-language links from localized body content are not allowed by default and require a written exception.

## 6. Phase Structure

### P1 - Technical Foundation

Build the routing, language configuration center, URL resolver, sitemap/hreflang system, preview rules, and publication gates.

P1 is successful only when the system can safely keep unfinished languages invisible to search engines.

See `docs/i18n/01-p1-technical-foundation.md`.

### P2 - Content Governance

Define the content model, admin workflow, translation states, review requirements, glossary rules, and page-class release criteria.

P2 is successful only when each page class has a clear path from source content to published localized content.

See `docs/i18n/02-p2-localization-content-governance.md`.

### P3 - LLM Localization Pipeline

Use LLMs to generate or assist localized content after P1/P2 are stable.

P3 is successful only when automation is cheaper and faster without bypassing terminology, structure, link, and search gates.

See `docs/i18n/03-p3-llm-localization-pipeline.md`.

---

## 7. Release Philosophy

The first language should be treated as a pilot, not as a bulk rollout.

Recommended pilot:

- Russian (`ru`)

Reasons:

- non-Latin language pressure tests routing, typography, and UI length;
- industrial B2B terminology needs glossary enforcement;
- enough search demand to justify careful localization;
- good test case for partial article release.

The pilot should publish a small but complete search-safe set:

- all required L1 pages;
- all required L2 navigation/catalog targets;
- selected L3 articles only after link audit.

After the pilot passes, additional languages should reuse the same gates.
